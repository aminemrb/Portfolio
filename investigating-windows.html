Investigating Windows

🖥️ Windows Server 2016
🔍 Digital Forensics & Incident Response
🔗 tryhackme.com/room/investigatingwindows

// Contexte

Dans ce challenge TryHackMe, je me retrouve face à une machine Windows Server 2016 qui semble avoir été compromise. Ma mission est de mener une investigation forensique complète en reconstituant la chaîne d'attaque — de l'accès initial à la mise en place du C2 — en utilisant uniquement les outils natifs Windows et les journaux d'événements disponibles sur la machine.

Étape 01

Version du système

▸Objectif Avant toute chose, j'ai besoin de connaître précisément l'environnement sur lequel je travaille : quelle version de Windows tourne sur cette machine ?
▸Méthode J'ouvre PowerShell et j'exécute la commande systeminfo. Cette commande native liste en détail les informations matérielles et logicielles du système, y compris le nom et la build de l'OS.
▸Résultat La sortie confirme que la machine tourne sous Microsoft Windows Server 2016 Datacenter, Build 14393.
Étape 02

Dernier utilisateur connecté

▸Objectif Je veux savoir quel compte utilisateur a été le dernier à ouvrir une session sur cette machine. Cela me donnera un premier indicateur sur qui contrôlait le système avant l'incident.
▸Méthode Je filtre l'Observateur d'événements sur le journal Sécurité en cherchant l'Event ID 4624, qui correspond à chaque ouverture de session réussie. Je trie les résultats par date décroissante pour isoler la toute dernière entrée.
▸Résultat La dernière connexion enregistrée est celle du compte Administrator, le 3/8/2026.
Étape 03

Dernière connexion de l'utilisateur John

▸Objectif En listant les comptes locaux, je repère un utilisateur nommé John. Je dois déterminer quand celui-ci s'est connecté pour la dernière fois, afin de le positionner dans la timeline de l'attaque.
▸Méthode Depuis PowerShell, j'exécute net user John | select-string "Last" pour extraire uniquement la ligne contenant la date de dernière connexion.
▸Résultat La dernière connexion de John remonte au 3/2/2019 à 5:48:32 PM. Cette date correspond précisément au début présumé de la compromission.
Étape 04

Persistance via le Registre

▸Objectif Après un accès initial, un attaquant cherche à maintenir sa présence même après un redémarrage. Je me concentre donc sur les mécanismes de persistance au démarrage, et notamment les clés de registre dites « Run ».
▸Méthode J'ouvre l'éditeur de registre (regedit) et je navigue jusqu'à la clé HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run. C'est l'un des emplacements classiques où les logiciels — ou les malwares — s'enregistrent pour s'exécuter automatiquement au démarrage de Windows.
▸Résultat Je découvre une entrée suspecte portant le nom BadrClient qui pointe vers un script avec une adresse IP codée en dur : 10.34.2.3. Il s'agit clairement d'un mécanisme de persistance posé par l'attaquant.
Étape 05

Analyse des privilèges administrateur

▸Objectif Je veux auditer le groupe des administrateurs locaux. Un attaquant ayant obtenu un foothold va souvent ajouter un compte à ce groupe pour garantir ses accès futurs sans passer par une élévation de privilèges.
▸Méthode J'exécute net localgroup administrators dans PowerShell pour lister tous les membres du groupe Administrateurs de la machine.
▸Résultat En plus du compte Administrator attendu, je retrouve dans ce groupe les comptes Guest et Jenny — deux comptes qui n'ont aucune raison légitime d'être administrateurs.
Étape 06

Tâche planifiée malveillante

▸Objectif Les tâches planifiées sont un autre vecteur de persistance très utilisé. Je cherche à identifier si l'attaquant en a créé une pour s'assurer d'une exécution régulière de son code malveillant.
▸Méthode J'ouvre le Planificateur de tâches Windows (Task Scheduler) et je parcours la bibliothèque de tâches, en cherchant des entrées anormales — noms suspects, déclencheurs inhabituels, ou actions pointant vers des scripts.
▸Résultat Je tombe sur une tâche au nom trompeur : Clean file system. Le nom est conçu pour passer inaperçu lors d'un audit rapide, mais ses propriétés révèlent une action malveillante.
Étape 07

Analyse du payload — Reverse Shell

▸Objectif Maintenant que j'ai identifié la tâche suspecte Clean file system, je dois déterminer exactement ce qu'elle exécute. L'action derrière une tâche planifiée malveillante est souvent le cœur du mécanisme de Command & Control.
▸Méthode Je clique sur la tâche dans le Planificateur et j'inspecte l'onglet Actions qui détaille les commandes ou scripts lancés.
▸Résultat L'action exécute nc.ps1 -l 1348 — un script PowerShell qui ouvre un reverse shell en écoute sur le port 1348, permettant à l'attaquant de reprendre la main sur la machine à tout moment.
Étape 08

Vérification de l'utilisateur Jenny

▸Objectif Le compte Jenny figure dans le groupe des administrateurs, mais est-il réellement utilisé ? Un compte fantôme créé par un attaquant et jamais utilisé reste néanmoins dangereux, car il peut servir de porte dérobée.
▸Méthode J'exécute net user Jenny pour consulter les informations de ce compte, notamment la date de dernière connexion.
▸Résultat Le champ Last Logon affiche Never : le compte a été créé, mais n'a jamais été utilisé pour ouvrir une session. C'est un compte de backdoor classique, mis en réserve par l'attaquant.
Étape 09

Date de la compromission initiale

▸Objectif La question centrale de toute investigation forensique est : quand l'attaque a-t-elle commencé ? Je dois établir la date précise du premier incident pour construire une timeline cohérente.
▸Méthode Je croise les horodatages collectés jusqu'ici : la date de création de la tâche planifiée malveillante, la date de dernière connexion de John, et les premiers événements suspects dans les logs de sécurité.
▸Résultat La corrélation des données pointe vers le 3/2/2019 comme date de compromission initiale — jour qui correspond à la fois à la dernière connexion de John et à la création des mécanismes de persistance.
Étape 10

Élévation de privilèges

▸Objectif Je cherche à identifier le moment exact où l'attaquant a réussi à obtenir des droits élevés sur le système. Ce pivot est un événement critique dans la chaîne d'attaque.
▸Méthode Je filtre les journaux de sécurité Windows sur l'Event ID 4672 — « Attribution de privilèges spéciaux à une nouvelle session ». Cet événement est généré à chaque fois qu'un compte avec des droits sensibles ouvre une session.
▸Résultat L'événement indique que des privilèges spéciaux ont été attribués à 4:04:49 PM le 3/2/2019, ce qui correspond à la fenêtre temporelle de la compromission initiale.
Étape 11

Outil de vol d'identifiants — Mimikatz

▸Objectif Une fois les privilèges élevés obtenus, l'attaquant cherche généralement à extraire des identifiants pour se déplacer latéralement dans le réseau. Je cherche à identifier quel outil a été utilisé à cette fin.
▸Méthode Je consulte les journaux de Windows Defender en filtrant sur l'Event ID 1116, qui correspond aux détections de malware par l'antivirus. Windows Defender peut avoir déclenché une alerte même si l'exécution a quand même eu lieu.
▸Résultat Windows Defender a détecté HackTool:Win32/Mimikatz.l sur le système. Mimikatz est l'outil offensif de référence pour extraire les hachages NTLM et les mots de passe en clair directement depuis la mémoire du processus LSASS.
Étape 12

Serveur C2 — DNS Poisoning

▸Objectif Je m'attelle maintenant à identifier l'infrastructure Command & Control (C2) de l'attaquant — le serveur distant depuis lequel il contrôle la machine compromise.
▸Méthode J'inspecte le fichier C:\Windows\System32\drivers\etc\hosts. Ce fichier est souvent modifié par des malwares pour rediriger le trafic DNS de domaines légitimes vers des serveurs contrôlés par l'attaquant, sans que la victime ne s'en aperçoive.
▸Résultat Le fichier a été falsifié : le domaine google.com est redirigé vers l'adresse IP 76.32.97.132, qui correspond au serveur C2 de l'attaquant. Toute requête vers Google depuis cette machine atterrit en réalité sur l'infrastructure pirate.
Étape 13

Découverte du Web Shell

▸Objectif Il me reste à identifier le vecteur initial d'accès — comment l'attaquant a-t-il mis un premier pied dans le système ? La machine étant un serveur web IIS, je concentre mon analyse sur le répertoire racine du serveur.
▸Méthode J'inspecte le contenu du répertoire C:\inetpub\wwwroot, qui héberge les fichiers publiés par IIS. Je recherche des fichiers inhabituels ou des scripts qui n'ont rien à faire dans cet espace.
▸Résultat Je découvre un fichier nommé b.jsp — un webshell Java Server Pages qui permet à quiconque connaît son URL d'exécuter des commandes arbitraires sur le serveur directement via HTTP, sans aucune authentification.
Étape 14

Ouverture de port dans le Pare-feu

▸Objectif Pour garantir la communication avec son infrastructure, l'attaquant doit s'assurer que le pare-feu ne bloque pas ses connexions. Je cherche des règles de pare-feu qui auraient été ajoutées sans autorisation.
▸Méthode J'ouvre le Pare-feu Windows Defender avec fonctions avancées de sécurité et j'examine les règles entrantes (Inbound Rules), en cherchant des règles récemment créées avec des noms ou des ports suspects.
▸Résultat Je trouve une règle frauduleuse intitulée "Allow outside connections for development", qui autorise toutes les connexions entrantes sur le port TCP 1337. Le nom est délibérément trompeur pour passer pour une règle légitime de développement.

// Synthèse — Chaîne d'attaque

1. Accès Initial (T1505.003) : L'intrusion commence par la compromission du serveur web où l'attaquant parvient à déposer un webshell b.jsp dans le répertoire IIS. Ce fichier lui offre une exécution de commandes à distance sans la moindre authentification.

2. Vol d'Identifiants (T1003) : Une fois l'accès initial sécurisé, l'outil Mimikatz est déployé sur le système pour extraire les hachages NTLM ou les mots de passe en clair directement depuis la mémoire du processus LSASS.

3. Élévation de Privilèges (Event 4672) : L'attaquant consolide son emprise en obtenant les droits d'administration à 4:04:49 PM le 3/2/2019. Il ajoute également les comptes Guest et Jenny au groupe Administrateurs locaux.

4. Persistance Multicouche (T1547.001 & T1053.005) : Pour survivre à un redémarrage, l'attaquant ancre sa présence à deux niveaux : la création d'une clé de registre BadrClient qui s'exécute automatiquement au démarrage, et la mise en place de la tâche planifiée Clean file system lançant un reverse shell via nc.ps1 sur le port 1348.

5. Command & Control (T1071) : La communication avec l'infrastructure pirate est assurée par un DNS poisoning du fichier hosts redirigeant le trafic vers l'IP 76.32.97.132, et par une règle de pare-feu ouvrant le port TCP 1337 aux connexions externes.
